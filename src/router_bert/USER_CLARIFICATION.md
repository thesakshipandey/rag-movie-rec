## What This BERT Router Does

### The Router is NOT User-Specific

**Important:** This BERT router is **prompt-based**, not user-based. It:

1. **Takes a prompt** (text description of what someone wants to watch)
   - Example: "I want a thrilling sci-fi movie with great special effects"

2. **Outputs weights** for 4 expert retrieval systems:
   - α (alpha): Weight for expert 1
   - β (beta): Weight for expert 2
   - γ (gamma): Weight for expert 3
   - δ (delta): Weight for expert 4

3. **Works for ANY user** - it's based on the prompt text, not user ID

### How It Relates to the 943 Users

The training data contains:
- **Prompts:** 1,000 text descriptions of movie preferences
- **Pairs:** 9,000 movie pairs (A vs B comparisons)
- **Expert scores:** For each pair, we have scores from 4 different retrieval experts

The prompts may have been generated by or for some of the 943 users, but the router learns to:
- **Understand the prompt text** (what kind of movies are being requested)
- **Route to the best experts** (which retrieval methods work best for this type of request)

### Example Flow

```
User 42 (from MovieLens) writes prompt:
  "Looking for a dark comedy with witty dialogue"
         ↓
BERT Router processes prompt text
         ↓
Outputs expert weights: [0.1, 0.3, 0.5, 0.1]
         ↓
Retrieval system uses these weights to combine 4 expert rankings
         ↓
Returns personalized movie recommendations for User 42
```

The router is **text-conditioned** (based on prompt) but the final recommendations are still **personalized** (for the specific user) because the expert systems themselves may use user history.

## Why This Design?

### Advantages of Prompt-Based Routing

1. **Generalizes to new users:** Works for users not in training data
2. **Adapts to context:** Same user gets different routing for different prompts
3. **Interpretable:** Can see which experts the router prefers for different types of requests
4. **Scalable:** One model serves all users

### Example Scenarios

**Scenario 1: User 42 wants action**
```
Prompt: "Fast-paced action movie with explosions"
Router: [0.05, 0.15, 0.70, 0.10]  # Prefers expert gamma for action
```

**Scenario 2: User 42 wants romance**
```
Prompt: "Heartwarming romantic comedy"
Router: [0.40, 0.35, 0.10, 0.15]  # Prefers experts alpha/beta for romance
```

**Scenario 3: New User 1000 (not in training)**
```
Prompt: "Mind-bending psychological thriller"
Router: [0.20, 0.25, 0.30, 0.25]  # Still works! Based on prompt text
```

## If You Need User-Specific Routing

If you want the router to consider user identity, you would need to:

1. **Add user embeddings** to the model
2. **Include user_id** in the input
3. **Modify architecture** to condition on both prompt + user
4. **Retrain** with user-aware data

But the current design is **prompt-only** and works well for this use case!

## Questions?

A: ALL of them! The router works for any user based on their prompt text.

**Q: Is the router personalized?**
A: The router itself is not user-specific, but the final recommendations are personalized because the expert retrieval systems use user history.
